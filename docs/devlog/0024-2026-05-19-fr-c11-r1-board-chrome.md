# 0024 — FR-C11 R1: paragon board chrome render model

*2026-05-19*

Owner ruling 2026-05-19 re-scopes board chrome from "consumer-owned,
not reproduced" (FR-C7 §6 / FR-C9 `WidgetRoles` "screen-chrome") to
library-decoded — "everywhere possible, use game data". CL-31 adds
the typed render model for board chrome and the no-drop gate FR-C11
asks for, with honest CL-28/CL-30-style discipline on the fire-border
art that turns out not to be authored in scene-data.

## What R1 settled

Scene-widget inspection in 657304 (`ParagonBoard`) and 964599
(`ParagonBoardSelect`) for each of FR-C11 R1's candidate handles:

- **`0x2954DF0C` (board background, 1200²)** — scene-bound on
  `Template_Board_Background_Center` in 657304. Authored rect is
  all-zero: the widget fills the parent canvas at native pixel size.
- **`0xDE8B9881`, `0x368C511E`** — scene-bound on `Board_BG`'s 0x58
  block in 964599 (the board-select panel's preview-frame backing).
  `Board_Icon_Template` binds the same handles per board instance.
- **`0x71C3ECC9`** — scene-bound on `Board_Icon_Filigrees` in 964599.
  A 1458×334 horizontal filigree band drawn around the board-select
  carousel.

Fire-border candidates:
- `0x6CFA1668`, `0x749F8139`, `0xAA7571AB` — in atlas SNO 447106
  (`2DUI_Paragon`) with frame sizes that look like fire-ember strip
  art when rendered, but **no widget in 657304 or 964599 binds them**.
- `0xB5C007F8` — bound, but to `Template_GlyphAura_Tile` (a glyph
  aura tile, not a board border).
- `0xC1473C21` — bound, but to `Common_Node_BG_Black` / `_Revealed`
  (per-node background art, not board overlay).
- `UI_Paragon_FrameGlow` (SNO 1364280, single-frame texture with
  handle `0x00000000`) — also engine-referenced; no scene binding.

So the fire-border animation is engine-internal: the engine
references those atlas frames directly and animates them; no scene
widget authors a frame sequence, order, or timing for the rim fire.
Per CL-28 / CL-30 discipline CASC does not fabricate a typed
sequence by guessing which catalog frames belong to the animation —
the typed model surfaces only scene-bound chrome.

## CL-31 — what shipped

1. **`ParagonBoardChrome` record** added to
   `WiseOwl.Casc.Diablo4`, surfacing:
   - `MainBoardBackground` (single `NodeElement`): scene 657304's
     background, scene-bound on `Template_Board_Background_Center`.
   - `BoardSelectChrome` (`IReadOnlyList<NodeElement>`): scene
     964599's `Board_BG` block handles + `Board_Icon_Filigrees`
     binding.

2. **`ParagonRenderModel.BoardChrome` field** — `ReadParagonRenderModel`
   now also builds and returns the typed chrome. Existing callers
   continue to access `Layout` + `Scenes` unchanged.

3. **`ParagonRenderProjection.BoardChrome`** — internal projection
   helper that pulls the chrome widgets by name from both scenes
   (`Template_Board_Background_Center` in 657304;
   `Board_BG` + `Board_Icon_Filigrees` in 964599) and populates
   `NodeElement` with atlas SNO + native px via the `FrameSize`
   lookup the existing layout/scene-model projections use.

4. **Tests:**
   - `ReadParagonBoardChrome_surfaces_scene_bound_chrome` asserts
     handle / atlas / native-px shape for the background and
     board-select chrome, and asserts no fire-border catalog handle
     leaks into the typed model.
   - `ReadParagonBoardChrome_layers_are_scene_bound` (gate) —
     parity with CL-26 / CL-27 / CL-28 / CL-30: every chrome layer's
     handle must appear in its scene's per-widget bindings.

5. **Spec §10.16** added (recipe + table + animation honest-report
   + acceptance). Appendix A CL-31 added.

Acceptance (live `3.0.2.71886`): 36/36 Diablo4 integration tests
green, including the new acceptance test and chrome scene-bind gate.

## Why FR-C7 §6 had excluded these (CL-28-grade)

`Project()` in `ParagonRenderProjection` projects per-node art (the
§7.2 19-row state matrix) — the per-rarity disc + ornate + overlay
states. CL-29's audit classified `Template_Board_Background_*`,
`Board_BG`, `Board_Icon_*`, `level_paragon_bg`, `Glyph_GridItem_*`,
etc. as "screen-chrome — out of node-icon scope" in the consumer's
`WidgetRoles`; CASC's projection mirrored that by not enumerating
board-chrome states. The widgets and their bindings have always been
present in the exhaustive `ReadParagonRenderModel().Scenes` view
(scene-bound, never dropped) — what was missing was the *typed*
projection. CL-31 adds it. No decode bug.

## Boundary

Library: decoded scene-bound chrome + scene-bind gate (CL-31).
Consumer: ImageSharp/runtime compositing, "21×21 board" rect mapping
(engine positions chrome at parent canvas native size; consumer
scales to runtime board-rect), and any procedural fire-border draw
or atlas-catalog walk for engine-internal art. FR-C7 §6 split is
preserved; only the scope widens to include board chrome.
