# 0054 — FR-C20: Catalog asset discovery/retrieval API

2026-05-21 · CL-55 (folds in CL-53) · branch `fr-c20-catalog-api`

## Trigger

Owner: *"(re)design our API so we can find a recipe, enumerate available
recipes (with filter), retrieve a recipe… The Optimizer should discover what it
needs dynamically, not hard code everything."* Then: *"future-proof to a
broader discovery/retrieval API. E.g. to get weapons, armor, etc."*

The library had grown a bespoke typed accessor per thing (`ReadParagonNodeRecipe`,
`ReadTiledStyle`, `ReadSelectionHighlight`, …). Good for known targets, useless
for discovery — the consumer had to hardcode SNO ids/names.

## Design decisions (owner-locked)

- **Scope:** render recipes + paragon domain, extensible to items/weapons.
- **Retrieval:** a *generic open* model — `TryGet(ref, out object)` /
  `TryGet<T>` returns the **existing decoded type** (the decoded records *are*
  the de-facto union; the consumer pattern-matches them). **No** closed
  `Recipe` wrapper hierarchy — that's exactly what couldn't absorb
  weapons/armor without editing the core every time. Plus keep the typed
  accessors as shortcuts.

## Shape

```
d4.Catalog
  Find(AssetQuery?) -> IEnumerable<AssetRef>   // lazy
  OfKind(kind) / TryResolve(kind, name, out ref)
  TryGet(ref, out object) / TryGet<T>(ref, out T) / TryGet(kind, sno, out object)

AssetRef(AssetKind Kind, SnoGroup Group, int Sno, string Name, IReadOnlyList<string> Tags)
AssetQuery { Kind?, Kinds?, NameContains?, Tag?, TagsAll?, RenderRecipesOnly, Where? }
```

Internals: an `IAssetProvider` registry, **one provider per `AssetKind`**, each
enumerating `CoreTOC.EntriesInGroup` and delegating decode to the existing
typed reader. Adding a family = one provider + one enum value, **zero facade
edits** — the future-proofing. `SnoProvider` (per-SNO, filter+tagger+decode
delegate), `SingletonProvider` (scene-projected recipes / fixed tables),
`SelectionHighlightProvider` (the atlas-source scan).

Kinds shipped: render (`ParagonNodeRender`, `TiledStyle`, `SelectionHighlight`,
`ParagonBoardGrid`, `TextureAtlas`); paragon domain (`ParagonBoard/Node/Glyph/
GlyphAffix`, `Power`, `PlayerClass`, `AttributeFormulas`); broader seed (`Item`,
`Affix` — proving extensibility).

## Exception-safety

`TryGet` never throws on bad game data — it catches the decode-failure family
(`CascException` and its subtypes, `FormatException`, `ArgumentException`,
`IOException`, `IndexOutOfRangeException`, `OverflowException`) and returns
`false`. Caught immediately by the leading `ParagonBoard` entry "Axe Bad Data",
a placeholder that fails the `Paragon_<Class>_<Index>` name convention.

## Folding in selection highlight (CL-53 / PR #39)

The standalone `ReadSelectionHighlight()` (CL-53, PR #39) is folded in: the
`SelectionHighlightProvider` discovers the shape-tagged selection TiledStyles,
and `ReadSelectionHighlight()` becomes a typed shortcut that walks
`Catalog.OfKind(SelectionHighlight)`. The public `SelectionHighlight` /
`SelectionHighlightStyle` / `SelectionShape` types are unchanged. **PR #39 is
superseded — to be closed unmerged.**

## Consumer usage (dynamic, no hardcoding)

```csharp
foreach (var r in d4.Catalog.Find(new() { Kind = AssetKind.Item, Tag = "weapon" }))
    if (d4.Catalog.TryGet<ItemDefinition>(r, out var weapon)) { … }

foreach (var r in d4.Catalog.Find(new() { Kind = AssetKind.SelectionHighlight }))   // shape in r.Tags
    if (d4.Catalog.TryGet<TiledStyleDefinition>(r, out var style)) { … }
```

## Acceptance

`Catalog_discovers_and_retrieves_assets_by_kind_filter` — singleton recipe
retrieval, shape-tagged selection find, `TryResolve`, name-filtered atlas,
domain-kind retrieval (skipping the bad-data placeholder), `RenderRecipesOnly`
exclusion, and the typed-shortcut agreement. 51/51 Diablo4 tests green on live
build `3.0.2.71886`.

## Next
