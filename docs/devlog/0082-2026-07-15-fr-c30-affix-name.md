# 0082 — FR-C30: first-party localized affix display name

2026-07-15 · CL-87 · branch `fr-c30-affix-name`

## Trigger

`casc-fr#42` from the Optimizer (Season 14, live build `3.1.1.72836`):
surface the **localized affix display name** as a first-class member,
mirroring the existing `AffixDefinition.Description`. Aspect display
names (`"Aspect of …"`) are absent from Maxroll's `data.min.json`
(desc-only) and from `CoreTOC.dat` (only the internal slug, e.g.
`578707 → legendary_barb_022`), so the library's sibling-StringList
route is the *only* first-party source of the name. The consumer had
verified it works against the prebuilt library — the ask is purely a
shipped surface so every consumer doesn't re-hand-roll the lookup.

## What was already there

`ReadAffix` (§11.3 / CL-22) already resolves the affix's sibling
`Affix_<snoName>` StringList table (group 42) — it just read the one
label `Desc`:

```csharp
TryReadSiblingString(SnoGroup.Affix, id, "Affix_", "Desc", locale, out var d);
a.SetDescription(d);
```

The **same table** carries a `Name` label (the display name) that was
never surfaced. Confirmed on the live build:

```
group 104  id  578755  Legendary_Barb_110
group  42  id 1106289  Affix_Legendary_Barb_110   [Name] = of Limitless Rage
group 104  id 1199626  Legendary_Barb_109
group  42  id 1199627  Affix_Legendary_Barb_109   [Name] = Devilish
group 104  id 2586362  Talisman_Charm_Affix_1HAxe_Unique_Generic_001
group  42  id 2586361  Affix_…                     [Desc] = Your attacks Critically Strike …   (no Name)
```

Coverage on `3.1.1.72836`: **1,464 / 6,145** group-104 affixes carry a
`Name`; the remainder are unnamed system/internal affixes.

## Fix

Pure sibling-label surface — no byte-layout change.

- `AffixDefinition.Name` — read alongside `Description` in `ReadAffix`
  (identical `TryReadSiblingString` mechanism, label `Name`), honest
  `string.Empty` sentinel when the affix has no `Name`.
- `Diablo4Storage.TryReadAffixName(int, out string, locale)` — the
  standalone name resolver, the affix analogue of the §6.4
  `TryReadParagonBoardName`: name-keyed via `CoreToc`, raw value only,
  honest `false`/`Empty` when absent. Lets a consumer resolve just the
  name without a full `ReadAffix` decode.

The raw authored fragment is returned verbatim (`"of Limitless Rage"`,
`"Bear Clan Berserker's"`, `"Devilish"`); the consumer owns any
`"Aspect …"` composition — consistent with `AffixDefinition` being
"identity + localized name and description only".

## Tests

Extended the live `C6_typed_readers_decode_identity_and_localized_text`
with positive + honest-empty anchors:

```csharp
Assert.Equal(string.Empty, aff.Name);                       // charm 2586362 = Desc-only
Assert.False(d4.TryReadAffixName(2586362, out _));
Assert.Equal("of Limitless Rage", d4.ReadAffix(578755).Name);   // Legendary_Barb_110
Assert.True(d4.TryReadAffixName(1199626, out var devilish));    // Legendary_Barb_109
Assert.Equal("Devilish", devilish);
Assert.Equal("Devilish", d4.ReadAffix(1199626).Name);
```

Passes on the live install `3.1.1.72836`.
