# 0107 — skill-tree API v1 (#57)

**Date:** 2026-07-18
**Work item:** casc-fr#57 (CASC proposal → Optimizer shaped it)
**CL:** CL-111 · `SkillTree` / `SkillTreeNode` + `Diablo4Storage.ReadSkillTree`

I proposed a skill-tree API (#57) rather than pre-building one; the Optimizer
shaped it: **logical tree, not rendering** — per-node kind / skill / modifier-group
/ effect + group membership + per-class, skip X/Y positions and edges (their KB is
markdown, not a renderer). This ships that MVP.

## Sources

- **`SkillTreeRewards` (g20/547685)** — the per-node metadata I fully decoded in
  phase-2: 284-byte records, tail F2 = modified-skill Power SNO, F4 = kind, F5 =
  modifier group id. Nodes filter to a class by name prefix (`Rogue_`, `Barb_`, …).
- **g39 board** — the `+0x10` VLA is the active-skill list ((skillSNO, flag) pairs).
- **Effect** — resolved through `SkillSno → ReadPower(...).Modifiers` (CL-104), per
  the Optimizer's "expose the SNO so I can join to text" request. No fiddly
  node→modifier-index join baked in; the consumer joins the way they join skills.

## Surface

```csharp
SkillTree t = d4.ReadSkillTree(SkillTreeClass.Rogue);
t.Nodes;                       // SkillTreeNode { Name, Kind, SkillSno, ModifierGroupId }
t.Skills;                      // active-skill Power SNOs
t.ModifierGroup(399111, 0);    // Blade Shift UpgradeA/B/C — the "pick one" set
```

Verified across all 8 classes; Blade Shift's two exclusive groups (UpgradeA/B/C =
group 0, Side1..4 = group 1) come out right, and the unlock→skill→`ReadPower`
effect join works.

## v1 scope (told to the Optimizer)

Logical structure only. **Deferred to v2:** the tier/category *cluster* grouping
(Basic / Core / … / Ultimate) — it needs the g39 board topology (position bands →
cluster), which is real work and I won't guess it. Also deferred: pinpointing a
node's *specific* modifier effect (v1 exposes the skill; the consumer reads its
modifier list). Both are iterate-from-feedback follow-ups, not blockers for the KB's
per-node reference.
