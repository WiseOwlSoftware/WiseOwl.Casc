## What R1 said vs what R2's broader anchor set reveals

R1 concluded the per-power stored value table was a clean 16-byte
slot record `[ASCII text][type=0x06][float][pad]`, with SF_N
mapping positionally for 3/4 anchored powers (Pyrosis / Overmind /
Ritualism) and Fathomless's SF_1 as a computed expression. The
trailing "10" universally a sentinel.

R2's broader scan across all 9 powers shows the picture is more
nuanced.

### Finding 1: SF_N is NOT always positional

**Dominion** is the counter-example. Owner labels: SF_0=0.5 (Wrath
cost reduction), SF_1=0.8 (damage buff), SF_2=12 (duration).
Stored slot table order: `[0.8, 0.5, 12, 10]`. The storage order
**does not match** SF_N positional order — SF_0 is at storage
position 1, SF_1 at position 0.

R1's 4 anchored powers all happened to be positional; that turned
out to be coincidence, not a general rule.

**Implication:** the SF_N → slot_index mapping must be defined
elsewhere in the blob — likely in the SF_N-named formula chains
(the ASCII `SF_N` identifiers in tail data, each preceding an AST
that resolves SF_N to a slot-index lookup or computed expression).

### Finding 2: Slot record structure varies between powers

**Ritualism** at 0x1720 has the clean R1 16-byte record:
`[ASCII@0..3][type=0x06@4..7][float@8..11][pad@12..15]`.

**Greater Hex** at 0x1738 has a different parse:
`[ASCII@0..3][zero-pad@4..7][type=0x06@8..11][float@12..15]`.

Either the records have variable internal byte order (unlikely for
a format spec) or there's a per-power preamble I haven't decoded
yet that shifts the alignment.

### Finding 3: SF_N ASCII labels are present sometimes, absent sometimes

In Dominion's tail data I find `SF_2` as an ASCII identifier
(0x1618) but NOT `SF_0` or `SF_1`. Either:
- SF_0 and SF_1 are inlined as trivial slot-index references
  without a named-AST entry.
- The named-AST entries use a non-ASCII identifier I haven't
  spotted yet.
- The labels are content-addressable via field-hashes rather than
  inline ASCII.

### Finding 4: Anchor byte-search misses for some new powers

Two new powers' anchors don't byte-match:
- **DemonicSpicules** SF_0=0.4 and SF_2=1.2: 0 hits each.
  SF_1=0.02 hits at 0x1A3C.
- **Overmind** SF_0=0.45 and SF_1=0.65: 0 hits each (R1 already
  noted: stored values are `0x3EE66667` / `0x3F266667`, one ULP
  higher than the owner-provided `0x3EE66666` / `0x3F266666`).

The Overmind case is well-understood (IEEE-754 round-to-nearest
vs round-toward-zero). DemonicSpicules' missing SF_0 and SF_2 may
be the same rounding pattern — needs verification.

## What's confirmed despite the complexity

- The 16-byte slot record stride and `(ASCII text form, type tag,
  float)` triple is structurally consistent. The internal byte
  order within the 16-byte record varies (Ritualism vs Greater
  Hex), but the triple is always present.
- The per-power tail-data slot table is the LAST contiguous run of
  16-byte slot records in the blob, terminated by `("0", 0.0)`.
  R1's seek-from-end strategy still works.
- The trailing "10" sentinel appears on every anchored power
  (owner R2 confirmed it as a hardcoded max-tier/rank cap, not a
  per-power SF_N value).
- Fathomless SF_1 = `SF_0 × stored_slot_1` = `0.15 × 7 = 1.05`
  computed expression — owner R2 explicit confirmation.

## What needs the next round to deliver option (b)

The AST decode is the central remaining work. To produce the
resolved `{SF_N → value}` map, the library must:

1. **Locate the SF_N → slot_index map** — either via the
   SF_N-named runs or another mechanism.
2. **Decode the AST opcode encoding** — at minimum: `LOAD_SLOT`,
   `ADD` (Ritualism's `1+SF_1`), `MUL` (Fathomless's
   `SF_0 × slot_1`), `MIN` (Demonic Spicules' cap).
3. **Per-power validation** against the owner's 9-power expected
   SF_N values (27 anchor pairs).

Tractable but non-trivial — at least one full session of focused
RE + implementation + test work, possibly two.

## Fallback option (a) is still viable as an interim

Owner's R2 message noted: *"Falling back to (a) if AST decode
stalls is straightforward (the stored slot table is delivered
either way)."* Option (a) would surface
`PowerDefinition.ScriptFormulaSlots` as a positional list of
`(text, float)` pairs — the stored slot table, decoded by
seeking from the end of blob. The consumer would then own the
SF_N → slot_index resolution + AST evaluation.

The R2 findings make option (a) less appealing than R1's
positional assumption suggested — without the SF_N → slot_index
map, the consumer can't directly use the stored slot table for
SF_N resolution. Option (a) would deliver a useful but incomplete
intermediate (the slot values + their ASCII text forms; consumer
still needs the AST chain for SF_N resolution).
