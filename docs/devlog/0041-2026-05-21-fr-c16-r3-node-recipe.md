# 0041 — FR-C16 R3: paragon node render program + the hierarchy answer (CL-44)

*2026-05-21*

Solves the blocker that FR-C16 + FR-C17 both bottomed out on: **how
the UI-scene blob encodes the widget hierarchy / draw order.** The
answer turned out to be simpler than a hidden parent-pointer field —
and it lets `ParagonNodeRecipe` ship now.

## The hierarchy question, answered

The decode hunt:

1. **No header hierarchy.** Every widget header's pre-class 16-byte
   region and post-sentinel words are **zero** across all 211 widgets
   in scene 657304. There is no parent-index / child-count / depth
   field in the per-widget header.
2. **No offset pointers.** No 4/8-byte value in any header window
   points at another widget's `nameStart`.
3. **Blob order IS the tree order.** The widgets are serialized
   depth-first in child order (the names confirm it:
   `ParagonNodes` → `ParagonNodes_BaseLayer` →
   `ParagonNodes_BaseLayer_VFX_Canvas`). Since UI children draw in
   child-array order, **blob order = draw / z-order.** The
   node-state-widget run (idx 96–130) is contiguous and in a sensible
   bottom-to-top composition (`Common_Node_BG_Black` → `Revealed` →
   glow → `IconBase` → rarity → `Purchased` → arrows → connectors).

So the "missing hierarchy" was never a separate field — the flat
widget list is already in z-order, and a subtree is a contiguous
sibling run. No full object-graph deserialization needed.

## What shipped

```csharp
public ParagonNodeRecipe Diablo4Storage.ReadParagonNodeRecipe();

public sealed record ParagonNodeRecipe(
    IReadOnlyList<ParagonNodeRecipeLayer> Layers);   // z-order

public sealed record ParagonNodeRecipeLayer(
    int    ZOrder,        // scene serialization order = draw order
    string WidgetName,    // engine state label, VERBATIM (predicate key)
    uint   WidgetClassId, // UIWindowStyle / UIBlinkerStyle / …
    uint   ImageHandle,   // hImageFrame (0 ⇒ runtime-bound / pure marker)
    WidgetRect Rect,
    byte   Alpha);        // dwAlpha
```

The node-template subtree is the contiguous run from the first
`Common_Node*` / `Node_*` widget through the last `Template_Node_*`
rarity sub-template, emitted in scene order. The consumer is a pure
interpreter:

```
for each node:
  for each layer L in recipe.Layers:       // already z-sorted
    if evaluate(L.WidgetName → predicate, node.runtimeState):
        draw(L.ImageHandle, L.Rect, L.Alpha)
```

## Predicate = name-keyed (no structural field)

R2 verified there is no per-widget state/predicate field — every
varying field is layout (`nWidth/nHeight/rect`), anchoring
(`eGroupType`, `eVerticalAnchoring`), or opacity (`dwAlpha`); the one
uncracked `DT_INT` `0x0CDB00E9` is not a state code. So the engine's
widget **name** is the state discriminator. Per
`feedback_widget-name-not-role`, CASC surfaces the name verbatim and
does NOT normalize it into a guessed role taxonomy; the consumer
carries the one explicit `name → runtime-predicate` binding (glue, not
composition invention — the composition itself is fully engine-sourced
here). If `0x0CDB00E9` later cracks as a state code, the surface
upgrades to structural with no consumer change.

## FR-C17 (board recipe) — same foundation

The board-level composition (#27) uses the identical principle: blob
order = z-order, board layers are a sibling run. The remaining FR-C17
gap is the grid **pitch** — not in the UI scene (the
`UIParagonBoardStyle` is a style wrapper with no grid-layout fields);
it is board-data-derived (the `.pbd` node grid coords × the engine's
cell metric). That's the next decode.
