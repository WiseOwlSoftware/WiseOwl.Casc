# 0040 — FR-C14 R9: UI tile-style typed surface (CL-42)

*2026-05-20*

Closes (provisionally) FR-C14 with a typed surface for the engine's
tile-rendering recipe — pivoting from "no API change, just RE" to
"land the API extension so the Optimizer can consume the
composition data". This devlog captures the cumulative-decode pass
that led here and the API shape that fell out of it.

## How we got here (the cumulative-decode win)

FR-C14 spent R1–R6 trying to identify the engine's mechanism for
rendering the paragon board's tiled background. After ruling out
multiple candidate paths (mip1 hypothesis, sibling-frame
compositing, `level_paragon_bg` widget), R6 reported the static-RE
wall: the widget class `0x112661D5` (Stack/Layout) holds
`0x22FF3AF6` in its `ExtraLayerValues` but the actual render recipe
is opaque under static binary analysis.

R7 applied two owner-relayed principles:

1. *"As we crack new hashes, we should go back and examine other
   data previously not interpretable."* — built a persistent hash
   dictionary at `docs/d4-hash-dictionary.md` + saved
   `feedback_cumulative-hash-decode`.
2. *"Public sources have some suggested / proposed names."* —
   discovered `blizzhackers/d4data` ships
   `!!D4FieldChecksums.yml`, a 197-KB precomputed registry of
   hash → name from upstream community RE.

The dictionary lookup against scene 657304's still-unknown field
hashes cracked the load-bearing one immediately:

## What `snoTiledStyle` points to

A `DT_SNO` field referencing a record in **SNO group 103**
(`SnoGroup.UiStyle`, format hash `0x80504E18`). The record type
hierarchy (per the `blizzhackers/d4data`
`!HorizontalTiledWindowPieces.5943238d.yml` schema, cited as
intel only) is `PolymorphicBase ← WindowPiecesBase ←
HorizontalTiledWindowPieces`:

```yaml
HorizontalTiledWindowPieces:
  +0x8   dwType         DT_UINT
  +0xc   dwPad          DT_UINT
  +0x10  flImageScale   DT_FLOAT
  +0x14  nPadding       DT_UINT
  +0x18  hPieceLeft     DT_UINT     # texture handle
  +0x1c  hPieceRight    DT_UINT
  +0x20  hPieceMiddle   DT_UINT
```

Verified record layout (8 dumped SNOs from scene 657304's
`snoTiledStyle` bindings):

```
+0x00  uint32   magic       = 0xDEADBEEF
+0x10  int32    SnoId       (self-reference)
+0x50  uint32   TypeTag     (polymorphic variant)
+0x58  float32  flImageScale
+0x60  uint32   PrimaryHandle (hPieceMiddle in the common variant)
+0x68..end variable-length per-variant suffix
```

Two TypeTag variants observed:

- `0xBC0D579E` — most common (7 of 8 dumped records).
  ImageScale 1.0 / 0.5 / 0.9, single primary handle, ~168-byte
  records. The schema-name backing this hash is not yet cracked.
- `0x02E46583` — observed on SNO 603760 `BagBackground` only.
  ~152-byte record (4 fewer bytes than the common variant);
  different trailing structure.

## What's still open (honest)

- `HasPartialDecode = true` on every 152/160/168-byte record —
  the trailing variant suffix (additional piece handles,
  per-piece sub-rects, padding spec) is undecoded. The community
  schema gives the *shape* (Left/Right/Middle for the horizontal
  variant; a hypothetical 9-slice would have all 8 edges + 4
  corners + 1 middle) but CASC has not first-party verified the
  byte offsets of those fields yet.
- TypeHash `0xBC0D579E` (the common variant's polymorphic tag) is
  uncracked. The cumulative-hash-decode principle will surface it
  on the next pass.
- The Optimizer's R5b "tile period + offset + wrap mode" question
  is partially answered (PrimaryHandle + ImageScale) but not
  fully (the explicit `nPadding` + per-piece UV rects need the
  variant-suffix decode).
- Whether `Vignette → InnerShadow` (843662) is *the* board-area
  pattern overlay the owner described remains owner-verifiable.
  CASC R9 surfaces the binding + parsed style; the visual
  cross-check is the consumer/owner's call.
