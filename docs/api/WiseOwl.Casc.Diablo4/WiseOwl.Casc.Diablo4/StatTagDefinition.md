# StatTagDefinition class

A decoded Diablo IV stat-threshold tag record (SNO group StatTag = 124). Surfaces the [`ThresholdFormulaText`](./StatTagDefinition/ThresholdFormulaText.md) only — evaluation (substituting the `ParagonBoardEquipIndex` binding and reducing the expression to a numeric threshold) is the consumer's, mirroring the [`AttributeFormulaTable`](./AttributeFormulaTable.md) boundary in `docs/casc-diablo4-format.md` Appendix C.

## Remarks

Group 124 names follow the pattern `<Stat>{Main|Side}{Tier}` (`WillpowerMain2`, `StrengthSide1`, …), with class-keyed composite variants (`Barb_Strength+Dexterity`) and glyph-keyed variants (`Glyph_Willpower_Main`). Two consumers reference these records:

* a rare [`ParagonNodeDefinition`](./ParagonNodeDefinition.md) via [`BonusStatTagSnoIds`](./ParagonNodeDefinition/BonusStatTagSnoIds.md) (the "bonus when threshold met" gating); and
* glyphs via their own activation gates (out of this CL's scope).

Byte layout (payload base `0x10`; verified on build `3.0.2.71886`): `snoId@0`; a `DT_VARIABLEARRAY[DT_CHAR]` descriptor at payload `+64` whose `dataOffset@+8` / `dataSize@+12` reference the ASCII formula text (no null counted in `dataSize`); a parallel `DT_VARIABLEARRAY` descriptor at payload `+80` carrying the pre-parsed token stream (not modeled — the text is the authoritative source; the token stream is the engine's bytecode equivalent).

Simple tags (`WillpowerMain2`, etc.) hold a single formula whose text sits at payload `+96`; class-composite tags (`Barb_Strength+Dexterity`) carry the primary formula at a later offset with additional structured sub-records for the per-alternative stats (not yet modeled — the canonical engine field names have not been recovered; surfacing the primary text only is the conservative first-cut). Glyph-keyed tags carry a numeric constant (`Glyph_Willpower_Main` → `"40"`) rather than a formula.
