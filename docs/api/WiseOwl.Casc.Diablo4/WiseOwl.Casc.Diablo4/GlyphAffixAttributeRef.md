# GlyphAffixAttributeRef structure

A single `(AttributeId, ParamPlus12)` entry from a glyph affix's [`AffectedAttributes`](./ParagonGlyphAffixDefinition/AffectedAttributes.md) list. The shape parallels the `ptAttributes` pair carried on a [`ParagonNodeDefinition`](./ParagonNodeDefinition.md) — an `eAttribute` int and a `uParamPlus12` GBID — but the glyph-affix encoding is the trimmed 8-byte pair only (no inline formula, no parallel attribute-GBID, no `NParam`), so it is surfaced as a distinct record struct rather than reusing [`NodeAttribute`](./NodeAttribute.md)'s 88-byte node-attribute shape.

```csharp
public struct GlyphAffixAttributeRef : IEquatable<GlyphAffixAttributeRef>
```

| parameter | description |
| --- | --- |
| AttributeId | The `eAttribute` int (matches the id used by [`GetAttributeName`](./Diablo4Storage/GetAttributeName.md) — names are resolved out of the [`LabelByAttributeId`](./AttributeNames/LabelByAttributeId.md) curated map or `AttributeDescriptions`; the AttributeId is itself a power-budget category, not a stat key — per the FR-C21 finding recorded in `casc-diablo4-format.md §7.6`). |
| ParamPlus12 | The associated GBID (`0xFFFFFFFF` when the attribute is tag-agnostic). On tag-conditional attribute ids — e.g. `AttributeId 259` (`DamageBonusTag`) — this GBID identifies the skill-tag the affix scales against (Abyss / Archfiend / Demonology / etc.); the consumer can call [`FormatFieldHash`](./Diablo4/FormatFieldHash.md) to render the raw GBID for uncracked names and [`HasParam`](./GlyphAffixAttributeRef/HasParam.md) to filter out the unset slots. |
