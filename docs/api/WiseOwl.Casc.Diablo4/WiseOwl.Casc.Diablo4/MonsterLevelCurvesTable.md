## Remarks

Correction. A prior CASC finding recorded this SNO as "an empty name-fragment registry, no per-tier curve." That was wrong — the curves are here. The record's `+0x50` array holds 6 × 320-byte tier records (`Raid_Tier_0..5` named inline), and each carries a `DT_VARIABLEARRAY` at record offset `+312` pointing to its curve rows in the record tail.

Curve shape (verified `3.1.1.72836`). Each tier's curve is a list of 12-byte rows = two `int32` + one `float32`. The two ints are equal in the live data (a level), and the float is the scaled value, which climbs from a small base to `100` across the tier's level span — e.g. Tier 0 spans levels 55→95 with scaled values reaching 100; higher tiers start at a higher base level (Tier 1 at 65, … Tier 5 at 105) and have fewer rows. The exact remap semantics (which int is input vs. band bound, whether the float is an effective level or a multiplier) are a structural inference, not owner-validated — so the row is exposed with descriptive names and the raw values ([`MonsterLevelCurvePoint`](./MonsterLevelCurvePoint.md)) so no data is hidden and no unverifiable formula is asserted (comprehensive-data-exposure + calibrate-claims-to-evidence).
