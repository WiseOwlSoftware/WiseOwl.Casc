## Remarks

Two separate curves (¬ß8.2 reconciliation). This is not the difficulty ladder (that lives in `StringList 216612`) and it is not the same curve as [`HpScalar`](./LevelScalingTable/HpScalar.md). The player-side `hpScalar` reaches √ó30.5 at level 70; this table's monster-HP column reaches √ó101,051 at level 70 ‚Äî a far steeper, independent curve (~3,300√ó different). Monsters scale their HP off this table, not off `LevelScaling` rows 71‚Äì200.

Label honesty (AC-3). The row-layout is locked by an independent anchor: [`PerLevelXpValue`](./DifficultyTiersTable/PerLevelXpValue.md) reproduces the game's per-level XP curve exactly (L40 = `8.0`, L70 = `11.0`; `+0.1`/level from ~L40) ‚Äî so the stride/offsets are correct and every column is located correctly. But [`MonsterHpScalar`](./DifficultyTiersTable/MonsterHpScalar.md) (col `+4`) and [`MonsterDamageScalar`](./DifficultyTiersTable/MonsterDamageScalar.md) (col `+8`) carry inferred semantics (devlog 0084): both are per-level multipliers (√ó1.0 at level 1, monotonic), but the "monster HP / damage" meaning cannot be owner-validated ‚Äî Diablo IV shows monster health as a bar only, with no numeric readout (owner-confirmed). Treat those two names as the best structural inference, not a verified label.

Byte layout (verified `3.1.1.72836`; ¬ß8.2): a `DT_VARIABLEARRAY` descriptor at payload `+0x50` (`dataOffset@+0` = 88 / `byteSize@+4` = 19200) ‚Üí 150 rows √ó 128 bytes. Columns: `+0``int32` level; `+4`/`+8` the HP/damage multipliers; `+36` the XP anchor; `+40` a per-level gold multiplier (candidate).
